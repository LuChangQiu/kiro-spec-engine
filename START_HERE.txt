1) Why kse is different
---------------------------------------------------------------

kse is not just "AI prompt helpers". It gives you:

- Structured delivery: Requirements -> Design -> Tasks
- Multi-Spec orchestration: run multiple specs in parallel safely
- Measurable outcomes: KPI snapshot/baseline/trend for weekly review
- Built-in governance: lock/doc/env/workspace controls for teams
- Tool-agnostic usage: works with Claude/Cursor/Windsurf/Copilot/Kiro


2) 90-second value proof
---------------------------------------------------------------

Run these commands in your project:

  kse adopt
  kse spec bootstrap --name 01-00-demo-feature --non-interactive
  kse value metrics sample --out ./kpi-input.json --json
  kse value metrics snapshot --input ./kpi-input.json --json

If command 4 succeeds, you get machine-readable KPI output with risk
signal and evidence paths.


3) Recommended workflow (single Spec)
---------------------------------------------------------------

  kse spec bootstrap --name 01-00-user-login --non-interactive
  kse spec pipeline run --spec 01-00-user-login
  kse spec gate run --spec 01-00-user-login --json


4) Recommended workflow (multi Spec / parallel)
---------------------------------------------------------------

  kse orchestrate run --specs "spec-a,spec-b,spec-c" --max-parallel 3
  kse orchestrate status
  kse orchestrate stop

Note: `kse spec bootstrap|pipeline run|gate run --specs ...` defaults
to orchestrate mode.


5) KPI observability quick commands
---------------------------------------------------------------

  kse value metrics sample --out ./kpi-input.json --json
  kse value metrics snapshot --input ./kpi-input.json --json
  kse value metrics baseline --from-history 3 --json
  kse value metrics trend --window 6 --json
